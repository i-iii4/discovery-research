# b2c-discovery-persona — JTBD-only discovery (v15)

**Суть проекта: разработка и калибровка методологии** поиска B2C возможностей через JTBD-интервью.
Это исследование метода, а не roadmap к немедленной разработке продукта.

Ветка `persona-experiment` переведена на `v15-jtbd-only`:
- без стартовых скаутов
- без web-search как обязательного входного фильтра
- две волны глубинных интервью (Discovery -> Validation)

## Required reading

- `methodology.md` — главный рабочий файл (`v15-jtbd-only`)
- `persona-protocol.md` — детальный протокол Wave A / Wave B
- `ARCHITECTURE.md` — схема стадий и контрольные правила
- `PLAN.md` — текущий план внедрения и калибровки v15
- `results.md` — исторические результаты запусков
- `DEVLOG.md` — журнал изменений

## Structure

```
b2c-discovery-persona/
  CLAUDE.md
  ARCHITECTURE.md
  PLAN.md
  DEVLOG.md
  methodology.md
  persona-protocol.md
  ajtbd-prompts.md
  results.md              # лог прогонов со ссылками на файлы
  hypotheses.base         # база данных гипотез (Obsidian Bases)
  jobs.base               # база данных работ (Obsidian Bases)
  pipeline.canvas
  jobs/                   # 1 файл = 1 работа (JTBD, спрос)
    Название работы.md    # frontmatter + 5 секций (формулировка, тип, связи, доказательства, workarounds)
  hypotheses/             # 1 файл = 1 бизнес-гипотеза (предложение, ссылается на работу)
    Название гипотезы.md  # v15 (AJTBD) и legacy (v2-v14), различаются полем methodology
  runs/
    v15-runN/
      interviews/
      round-*.md
```

### Два слоя: работы и гипотезы

- **Работа** (`jobs/`) — атомарная единица спроса. Формулировка JTBD, тип по AJTBD, уровень, связи с другими работами, доказательства из интервью, текущие workarounds. Живёт независимо от конкретного прогона.
- **Гипотеза** (`hypotheses/`) — предложение (решение). Бизнес-модель, цена, риски, тесты. Ссылается на работу через `job:` в frontmatter. Привязка к прогону — через поле `run:` в frontmatter.

## Запуск пайплайна (v15)

```
1. Обновить профиль основателя и ограничения в methodology.md
2. Создать директорию runs/v15-runN/
3. Создать runs/v15-runN/checklist.md
4. Запустить R0-R5 по methodology.md
5. Обновить results.md + DEVLOG.md
```

### Команда для запуска

`Запусти B2C discovery v15-jtbd-only по methodology.md с полным чеклистом стадий.`

### Управление контекстом (КРИТИЧНО)

Пайплайн выполняется в одной сессии (R0 — R5). Контекст контролируется через правило агентов:

**Каждый агент обязан записать результат в файл и вернуть только статус (путь + строк + 2-3 предложения). Полный текст в контекст оркестратора не возвращать.**

## Методологическая культура

### Запрещено

- Пропускать Wave B (R4) и публиковать ALIVE как валидные
- Переиспользовать персон из Panel A в Panel B
- Делать manual override без `override_reason`
- Менять >3 переменных между соседними прогонами
- Оценивать гипотезы "на глаз" вне структуры пайплайна
- Предлагать "перейти к прототипу" или "начать строить MVP" (это вне scope)
- Возвращать полный текст артефакта из агента в ответ оркестратору (вместо записи в файл + статус)

### Обязательно

- Для каждого запуска хранить отдельные артефакты `round-*` в `runs/v15-runN/`
- Хранить интервью по путям:
  - `runs/v15-runN/interviews/wave-a/`
  - `runs/v15-runN/interviews/wave-b/`
- Поддерживать консистентность между `runs`, `results.md`, `DEVLOG.md`
- Явно фиксировать правило выбора домена на R0
- Обновлять checklist по мере прохождения стадий
- Хранить полные интервью в файлах, в контекст передавать только статус (см. Context budget protocol)
- Агент = Write файл + вернуть статус. Не возвращать полный текст артефакта
- В каждой расшифровке сначала `## Summary`, потом `## Transcript`
- Для всех интервью использовать шаблон `interview-template.md`

## Язык

Весь текст — на русском. Это касается всех агентов (Generator, Interviewer, Scoring, Synthesis).

**На русском обязательно:**
- Названия jobs и гипотез («Объяснение метрик для клиентов», не «Metric Contextualization»)
- Job statements («Когда мне нужно...», не «When I need to...»)
- Аналитический текст, вердикты, обоснования
- Цитаты персон (персоны говорят по-русски)
- Названия кластеров, сегментов, категорий

**Английский допустим только для:**
- Имён собственных продуктов и компаний (Notion, Zapier, Intercom, Stripe)
- Устоявшихся терминов без адекватного перевода (JTBD, MRR, churn, SaaS, API)
- Идентификаторов (JOB-01, H5, pers01)
- Кода и команд терминала

**Запрещён рунглиш** — необоснованные вставки английских слов:
- «breadth» → «охват», «severity» → «острота», «switching signal» → «сигнал переключения»
- «churned» → «отказался от», «capped» → «ограничен», «override» → «ручная корректировка»
- «discovery» → «обнаружение», «validation» → «валидация» (устоявшийся термин, допустим)

## Scope notes

- Проект исследует качество методологии, а не занимается production build.
- Допустимы `WEAK`/`KILL` результаты, если пайплайн валиден и воспроизводим.
- Главный KPI: качество разделения jobs между Wave A и независимой Wave B.
