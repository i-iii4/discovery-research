# B2C Pet-Project Discovery v15-jtbd-only

> Методология мультиагентного поиска B2C jobs через JTBD-интервью.
> Фокус: 2 недели до MVP, solo dev, без обязательного web-search на входе.
> Runtime: Claude Code (Task + TeamCreate).
> Ветка: `persona-experiment`.

## Что изменилось в v15

- Удалены стартовые Scout-агенты и сигнальный веб-поиск.
- Вход теперь начинается с `Domain Seed` (предметная область + сегменты).
- Валидация строится на двух независимых волнах глубинных интервью:
  - Wave A: discovery jobs + workaround chains
  - Wave B: независимая проверка выбранных jobs
- Пайплайн стал `JTBD-only`: веб-поиск больше не является обязательным источником спроса.

### Доработки после run4 (iGaming post-mortem)

Run4 показал слепое пятно: протокол находил только операционные проблемы (антагонистичные инкумбенту работы), но пропускал ценностные работы (совпадающие с бизнес-моделью инкумбента). Три изменения:

1. **R0 — Ландшафт предложения.** Обязательная секция: тип домена (закрытый/открытый/гибридный), инкумбенты, зоны совпадения и антагонизма. Контекст для фильтра на R2.5, не передаётся интервьюеру.
2. **R2 — Блок 6 «Ценностные работы».** Новый обязательный блок интервью: лучший опыт, нереализованные ожидания, реальная готовность платить. Компенсирует системный перекос блоков 1-5 в сторону боли.
3. **R2.5 — Фильтр совпадения интересов.** Каждая работа получает тег `alignment` (совпадающая/нейтральная/антагонистичная). Правило отсечения зависит от типа домена: в закрытых платформах антагонистичные работы не проходят в shortlist, в открытых рынках — проходят как возможности для дизрапции.

### Доработки после run5 (web-saving post-mortem)

Run5 выдал 0 ALIVE / 7 WEAK. Причина: R0 описал домен через действие + механику («сохранение информации для повторного использования»), что ограничило весь пайплайн функциональным слоем. Эмоциональные, социальные и идентичностные работы (думать через коллекционирование, выражать вкус, делиться перспективой) не были обнаружены. Четыре изменения:

1. **R0 — Карта слоёв опыта.** Домен обязательно раскладывается по 4 слоям (функциональный, эмоциональный, социальный, идентичностный). Под-сегменты покрывают минимум 3 из 4.
2. **R1 — Проверка разнообразия персон.** Минимум 2 персоны с непрофессиональным отношением к домену. Минимум 1 персона, для которой домен — часть идентичности.
3. **R2 — Техника цепочек поведения.** Во всех блоках интервью: для каждого действия — вверх (зачем?), вниз (что дальше?), назад (что до?), вперёд (когда вернулся?). Вскрывает иерархию работ в интервью, а не аналитически постфактум.
4. **R2.5 — Обязательный граф работ.** После синтеза — иерархия: parent_job, child_jobs, проверки полноты (пропущенные обеспечивающие работы, покрытие слоёв, кластерные сигналы).

## Главная идея

Скауты исторически давали две функции: рандомизацию и начальную заземлённость.
В v15 рандомизация переносится в явный выбор домена, а заземлённость в интервью
реальных контекстов (backward-looking, Mom Test).

## Pipeline map

1. `R0 Domain Seed`
2. `R1 Persona Panel A (Discovery)`
3. `R2 Deep Interviews A`
4. `R2.5 Job Synthesis`
5. `R3 Persona Panel B (Validation)`
6. `R4 Deep Interviews B (two-agent mandatory)`
7. `R4.5 Panel Scoring + RAT`
8. `R5 Synthesis`

## Stage details

### R0 — Domain Seed

**Цель:** задать контролируемую область исследования без скаутов.

**Вход:**
- профиль основателя
- anti-domains
- backlog discovered jobs из прошлых прогонов (если есть)

**Выход:**
- `domain`: 1 предметная область
- `subsegments`: 2-3 конкретных сегмента (по слоям опыта, см. ниже)
- `domain_reason`: почему этот выбор сделан сейчас
- `exclusions`: что явно исключено
- `experience_layers`: карта слоёв опыта (см. ниже)
- `supply_landscape`: ландшафт предложения (см. ниже)

**Карта слоёв опыта (обязательная секция):**

Домен раскладывается по четырём слоям. Для каждого — примеры работ и типичные персоны:

| Слой | Вопрос | Что ищем |
|------|--------|----------|
| Функциональный | Что делает? | Операционные работы: сохранить, найти, организовать |
| Эмоциональный | Что чувствует? | Работы-состояния: спокойствие, вдохновение, контроль, азарт |
| Социальный | Как взаимодействует? | Работы-коммуникации: показать, поделиться, соавторствовать |
| Идентичностный | Кем себя ощущает? | Работы-самоопределения: быть куратором, экспертом, творцом |

**Правило покрытия:** под-сегменты обязаны покрывать **минимум 3 из 4 слоёв**. Если все сегменты описывают варианты одного слоя (например, три способа сохранять) — R0 недостаточен и требует доработки.

**Анти-паттерн:** домен, описанный через действие + механику («сохранение информации для повторного использования»), порождает только функциональные работы. Описание должно включать опыт и смысл («коллекционирование, мышление и самовыражение через сохранённый контент»).

**Ландшафт предложения (обязательная секция):**

Для каждого домена определить:

1. **Тип домена** — контролирует ли инкумбент доступ пользователя к продукту?
   - *Закрытая платформа* — инкумбент контролирует среду (iGaming, маркетплейсы, экосистемы)
   - *Открытый рынок* — пользователь свободен вне продукта инкумбента (путешествия, финансы, хобби)
   - *Гибридный* — зависит от конкретной работы
2. **Инкумбенты** — кто сейчас обслуживает этот спрос
3. **Бизнес-модель инкумбентов** — за счёт чего зарабатывают
4. **Зоны совпадения** — категории работ, которые усиливают бизнес-модель инкумбента
5. **Зоны антагонизма** — категории работ, которые ослабляют бизнес-модель инкумбента

Карта используется на R2.5 при синтезе работ. Интервьюеру на R2 она **не передаётся** — чтобы не вносить предвзятость в обнаружение.

**Правила:**
- не использовать broad domain вроде "productivity" без сужения
- минимум один сегмент должен быть non-obvious
- фиксировать рандомизацию выбора (seed/правило отбора)

**Артефакт:** `runs/v15-runN/round-0-domain-seed.md`

### R1 — Persona Panel A (Discovery)

**Цель:** собрать панель персон для выявления работ.

**Размер панели:** 8-10 персон

**Распределение:**
- 3-4 HIGH intent
- 3-4 MEDIUM intent
- 2 LOW intent
- минимум 2 churned-персоны

**Требования к персоне:**
- роль и контекст
- последний эпизод, когда работа возникла
- текущий workaround (конкретные шаги)
- фактические затраты (время/деньги)
- история переключения
- доминирующий слой опыта (функциональный / эмоциональный / социальный / идентичностный)

**Проверка разнообразия (обязательна):**
- Панель должна покрывать минимум 3 из 4 слоёв опыта из R0
- Запрещено: все персоны — профессионалы с утилитарным контекстом
- Минимум 2 персоны с непрофессиональным отношением к домену (хобби, самовыражение, сообщество)
- Минимум 1 персона, для которой домен — часть идентичности, а не инструмент

**Артефакт:** `runs/v15-runN/round-1-personas-A.md`

### R2 — Deep Interviews A (Discovery)

**Цель:** извлечь jobs, триггеры, барьеры, альтернативные объяснения.

**Формат:**
- глубинные интервью по персоне
- backward-looking only
- 2 интервью-команды параллельно

**Обязательные блоки:**
1. Last incident
2. Current workaround
3. Switching behavior
4. Priority ranking in real life
5. Adjacent jobs discovery
6. Value jobs (ценностные работы)

**Блок 6 — Value jobs (обязателен):**

Блоки 1-5 ориентированы на трение и боль — они систематически находят операционные проблемы, но пропускают ценностные работы (удовольствие, самовыражение, принадлежность). Блок 6 компенсирует эту слепую зону.

Вопросы:
- Опиши свою лучшую сессию / опыт за последний месяц. Что именно сделало её хорошей?
- Что бы ты хотел получить от [продукта/сервиса], чего сейчас не получаешь?
- За что ты реально готов платить больше? (не «купил бы», а «уже платишь» или «переплачиваешь»)

В промпт интервьюера добавляется инструкция: «Исследуй обе зоны — и трение, и ценность. Не фокусируйся только на проблемах.»

**Техника цепочек поведения (применяется во всех блоках):**

Для каждого действия или работы, упомянутых респондентом, интервьюер обязан проследить цепочку вверх и вниз:

- **Вверх (зачем?):** «Зачем ты это сохранил? Какую задачу решал? Что было бы, если бы не сохранил?»
- **Вниз (как?):** «Что ты сделал дальше? Куда положил? Что произошло потом?»
- **Назад (что предшествовало?):** «Что произошло перед этим? Откуда пришла ссылка? Что ты делал в тот момент?»
- **Вперёд (что следует?):** «Когда ты вернулся к этому? В каком контексте? Что делал с сохранённым?»
- **Без (что если нет?):** «Что бы случилось, если бы ты не смог это сделать?»

Цель — вскрыть иерархию: действие → обеспечивающая работа → основная работа → высокоуровневая цель. Не фиксировать только то, что респондент сказал буквально, а прослеживать всю поведенческую цепочку.

**Выход на персону:**
- подтверждённые jobs
- **цепочки поведения** (job chains: действие → зачем → зачем → корневая работа)
- severity/frequency
- existing spend
- switching trigger/barrier
- quotes
- discovered jobs (outside initial scope)

**Артефакты:**
- индекс по волне: `runs/v15-runN/round-2-interviews-A.md`
- полные расшифровки: `runs/v15-runN/interviews/wave-a/*.md`

### R2.5 — Job Synthesis

**Цель:** превратить сырые интервью в рабочий shortlist jobs.

**Процедура:**
- дедуп jobs
- кластеризация по контексту и outcome
- ранжирование по Breadth/Severity/Frequency
- оценка совпадения интересов (alignment) — см. ниже
- создание атомарных файлов работ в `jobs/`

**Фильтр совпадения интересов (обязателен):**

Для каждой обнаруженной работы ответить на вопрос: «Если эта работа будет решена, инкумбент заработает больше, меньше или столько же?»

Каждая работа получает в frontmatter:
```yaml
alignment: совпадающая / нейтральная / антагонистичная
alignment_reason: "Краткое обоснование"
```

Правило отсечения зависит от типа домена (определяется на R0):

| Тип домена | Антагонистичная работа | Действие |
|------------|----------------------|----------|
| Закрытая платформа | Инкумбент может заблокировать решение | Каталог обнаруженных работ, **не в shortlist** |
| Открытый рынок | Инкумбент не контролирует пользователя | В shortlist с пометкой «дизрапция» |
| Гибридный | Зависит от механизма блокировки | Оценить конкретный механизм, решение по каждой работе |

В файле `round-2.5-job-synthesis.md` добавляется таблица фильтра совпадения.

**Граф работ (обязателен):**

После синтеза плоского списка — построить иерархию. Для каждой работы ответить:
- **Зачем?** (upstream) → родительская работа (`parent_job` в frontmatter)
- **Как?** (downstream) → дочерние/обеспечивающие работы
- **Уровень** → `основная` (самостоятельная ценность) или `вспомогательная` (обеспечивает другую работу)

Проверки полноты:
1. **Кластерный сигнал:** если 3+ листовых работ ведут к одному родителю — родитель должен быть в shortlist (даже если не назван явно)
2. **Пропущенные обеспечивающие работы:** если есть «сохранить надёжно», где «экспортировать»? где «восстановить»? Очевидные дочерние работы добавляются в shortlist
3. **Покрытие слоёв:** все ли слои опыта из R0 представлены в графе? Если нет функциональных — проблема. Если нет эмоциональных — тоже проблема
4. **Орфанные работы:** листовая работа без родителя — слабый сигнал, требует обоснования

В файле `round-2.5-job-synthesis.md` добавляется секция «Граф работ» с визуальной иерархией (текстовое дерево или таблица).

**Выход:**
- 8-12 jobs для Wave B (после фильтра совпадения и проверки графа)
- Атомарные файлы работ (`jobs/Название работы.md`) со стадией `синтезирована`
- Каждый файл работы содержит: формулировку JTBD, тип по AJTBD, доказательства Wave A, workaround-таблицу, тег alignment, parent_job и level

**Два слоя:**
- **Работа** (`jobs/`) — чистый спрос: формулировка, тип, уровень, связи, доказательства
- **Гипотеза** (`hypotheses/`) — предложение: решение, цена, риски, тесты → ссылается на работу через `job:` в frontmatter

**Артефакт:** `runs/v15-runN/round-2.5-job-synthesis.md` + файлы в `jobs/`

### R4 — Deep Interviews B (Validation)

**Цель:** независимая проверка shortlist jobs.

**Формат:**
- строго двухагентный (TeamCreate):
  - Interviewer знает job hypothesis
  - Persona не знает hypothesis
- 2 команды параллельно

**Обязательные блоки:**
1. Повторяемость проблемы (real episodes)
2. Текущий процесс и friction points
3. Готовность к смене процесса (по прошлому опыту)
4. Цена без вопроса "купил бы" (через revealed preferences)
5. Альтернативные приоритеты (чем job проигрывает)

**Блокер:**
- если R4 пропущен, run получает статус `UNVALIDATED`

**Артефакты:**
- индекс по волне: `runs/v15-runN/round-4-interviews-B.md`
- полные расшифровки: `runs/v15-runN/interviews/wave-b/*.md`

### R4.5 — Panel Scoring + RAT

**Score 0-100:**
- Breadth (3x)
- Severity (2x)
- Frequency (2x)
- Current Spend (1x)
- Switching History (1x)

**Thresholds:**
- ALIVE >= 60
- WEAK 40-59
- KILL < 40

**RAT (P x I):**
1. demand fragility
2. economic segment quality
3. value capture / WTP
4. unit economics
5. scale constraints

**Risk rule:**
- любой риск > 20 = RED FLAG
- RED FLAG должен иметь тест mitigations (до 3 проверок)

**Артефакт:** `runs/v15-runN/round-4.5-scoring-rat.md`

---

### R5 — Synthesis

**Выход 1 (обязательный):**
- shortlist jobs с verdict (`ALIVE/WEAK/KILL`)
- top-3 обоснования по каждому verdict
- red flags + mitigation tests

**Выход 2 (обязательный):**
- `Discovered Jobs Catalog` для следующего цикла
- кластеры + приоритет + из каких интервью поднят сигнал

**Выход 3 (обязательный):**
- Отдельные файлы гипотез в `hypotheses/`
- Формат: 1 файл = 1 гипотеза (frontmatter + секции решения/рисков/тестов, см. `persona-protocol.md`)
- Каждая гипотеза ссылается на работу через `job:` в frontmatter
- Группировка по статусу через `hypotheses.base`
- Обновление `results.md` в формате лога со ссылками на файлы

**Выход 4 (обязательный):**
- Обновление атомарных файлов работ в `jobs/`:
  - Стадия → `оценена`
  - Вердикт → `ALIVE`/`WEAK`/`KILL`
  - Добавить доказательства Wave B
  - Обновить `validated_in` ссылками на интервью Wave B

**Артефакты:**
- `runs/v15-runN/round-5-synthesis.md` — сводка + каталог обнаруженных работ
- `hypotheses/*.md` — карточки гипотез (по одной на работу)
- `jobs/*.md` — обновлённые файлы работ (стадия, вердикт, доказательства Wave B)
- `results.md` update (лог со ссылками)
- `DEVLOG.md` update

## Prompts (core)

### Prompt: Domain Seed

```text
Ты — Domain Seeder для JTBD-only discovery.

Input:
- Founder profile
- Anti-domains
- Previous discovered jobs backlog (if any)

Task:
1) Выбери 1 предметную область и 2-3 под-сегмента.
2) Разложи домен по 4 слоям опыта (функциональный, эмоциональный, социальный, идентичностный).
   Для каждого слоя — примеры работ и типичные персоны.
3) Зафиксируй правило выбора (почему именно сейчас).
4) Определи исключения (что не исследуем в этом цикле).
5) Сформируй scope statement (1 абзац, включающий не только действие, но и опыт/смысл).
6) Составь ландшафт предложения:
   - Тип домена: закрытая платформа / открытый рынок / гибридный
     (критерий: контролирует ли инкумбент доступ пользователя к продукту?)
   - Инкумбенты и их бизнес-модель
   - Зоны совпадения интересов (работы, усиливающие бизнес-модель)
   - Зоны антагонизма (работы, ослабляющие бизнес-модель)

Rules:
- Нельзя брать слишком общий домен.
- Нужен минимум один non-obvious сегмент.
- Под-сегменты обязаны покрывать минимум 3 из 4 слоёв опыта.
- Выход должен быть пригоден для создания персон.
- Ландшафт предложения — обязательная секция, используется на R2.5.
- Анти-паттерн: описание домена через действие + механику (напр. «сохранение для использования»).
  Описание должно включать опыт и смысл.
```

### Prompt: Persona Builder (A/B)

```text
Создай панель персон для JTBD-интервью.

Input:
- Domain scope
- Goal: DISCOVERY (A) или VALIDATION (B)

Output:
- 8-10 персон
- Для каждой: role, context, trigger, last incident, workaround,
  spend, switching history, intent level

Rules:
- Панель должна быть разнообразной по контексту и intensity.
- Для Wave B нельзя повторять персон из Wave A.
- Никаких абстрактных профилей; только конкретные контексты.
```

### Prompt: Deep Interview (A)

```text
Проведи backward-looking JTBD-интервью.

Goal:
- обнаружить реальные jobs, а не подтвердить гипотезу

Blocks:
1) Last incident
2) Current workaround steps
3) Switching behavior
4) Priority in real life
5) Adjacent jobs (outside scope)
6) Value jobs — ценностные работы:
   - Опиши лучшую сессию/опыт за последний месяц. Что сделало её хорошей?
   - Что хочешь получить от [продукта/сервиса], чего сейчас не получаешь?
   - За что реально готов платить больше? (не «купил бы», а «уже платишь/переплачиваешь»)

Техника цепочек поведения (ПРИМЕНЯТЬ ВО ВСЕХ БЛОКАХ):
Для каждого действия или работы, упомянутых респондентом, проследи цепочку:
- Вверх: «Зачем? Какую задачу решал? Что было бы, если бы не смог?»
- Вниз: «Что сделал дальше? Куда положил? Что произошло потом?»
- Назад: «Что было перед этим? Откуда пришло? Что делал в тот момент?»
- Вперёд: «Когда вернулся к этому? В каком контексте?»
Цель — вскрыть иерархию: действие → обеспечивающая работа → основная работа → цель.

Важно: исследуй обе зоны — и трение, и ценность. Не фокусируйся только на проблемах.

For each interview return:
- confirmed jobs (pain + value)
- job chains (действие → зачем → зачем → корневая работа)
- severity/frequency
- current spend
- switching trigger/barrier
- quotes
- discovered jobs
```

## Context budget protocol (mandatory)

### Правило 1: файлы на диск, статус в контекст

- Каждый агент **сначала записывает результат в файл** (Write tool), затем возвращает только статус.
- Допустимый формат возврата агента в контекст оркестратора:
  ```
  Файл записан: runs/v15-runN/round-X-xxx.md (NNN строк).
  Краткий итог: [2-3 предложения].
  ```
- Запрещено возвращать полный текст артефакта в ответе агента.
- Полные транскрипты интервью сохраняются в:
  - `runs/v15-runN/interviews/wave-a/` для Wave A
  - `runs/v15-runN/interviews/wave-b/` для Wave B

### Правило 2: промпт агента обязан содержать инструкцию записи

Каждый промпт для Task-агента должен заканчиваться блоком:

```
ОБЯЗАТЕЛЬНО:
1. Запиши полный результат в файл [путь] через Write tool.
2. Верни мне ТОЛЬКО: путь к файлу, количество строк, и краткий итог (2-3 предложения).
3. НЕ возвращай полный текст документа в ответе.
```

### Правило 3: контроль контекста

Пайплайн выполняется в одной сессии (R0 — R5). Контекст контролируется через правило агентов: каждый агент записывает результат в файл и возвращает только статус.

Между стадиями оркестратор:
1. Убеждается, что все файлы записаны на диск
2. Обновляет checklist.md
3. Продолжает со следующей стадии
