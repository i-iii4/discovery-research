## Phase 13 — обогащение пайплайна методологией AJTBD [FUTURE]

### Контекст

Сравнение текущего пайплайна v15 с полным курсом Advanced Jobs to Be Done (Замесин) выявило: проект хорошо реализует нижнюю часть AJTBD (формула работы, backward-looking подход, иерархия Big/Core/Small/Micro, сегментация по работам), но недоиспользует верхнюю часть — граф работ как единицу анализа, типизацию работ, механики создания ценности, веральность и ориентационные работы.

### Что уже есть (не трогаем)

| Концепция AJTBD | Реализация в v15 | Оценка |
|---|---|---|
| Иерархия работ (Big → Core → Small → Micro) | ajtbd-prompts.md, scoring | Хорошо |
| Формула работы (когда/хочу/чтобы) | job_statement в синтезе и персонах | Хорошо |
| Backward-looking интервью (только прошлое) | Принцип #1 persona-protocol.md | Хорошо |
| Ложные работы (фильтр фантазий) | «Только факты прошлого» | Хорошо |
| Workaround chains | Блок 2 интервью | Хорошо |
| Switching behavior | Блок 3 интервью (триггер/барьер) | Хорошо |
| Сегментация по работам, а не демографии | Промпты B2C/B2B | Хорошо |
| 3 вопроса реальной сегментации | RAT overlay + ajtbd-prompts | Хорошо |
| Критическая последовательность работ (КПР) | ajtbd-prompts, косвенно в scoring | Средне |
| Consideration set | Косвенно через switching behavior | Слабо |
| Активирующие знания | Не формализовано | Отсутствует |

### Что отсутствует (разрывы)

#### Разрыв 1: Плоский список вместо графа работ

Работы обнаруживаются и оцениваются изолированно (Breadth, Severity, Frequency). Связи между ними не строятся. R2.5 Job Synthesis делает кластеризацию, но кластер — это не граф. Граф показал бы:
- какая работа выше уровня объединяет несколько Core Jobs
- какие работы ниже уровня порождает каждый workaround
- какие работы последовательны (КПР)
- какие связи «многие ко многим» (одна Small Job служит нескольким Core Jobs)

**Ценность**: продукт для связки работ ценнее, чем для одной изолированной. Граф покажет, что несколько WEAK-jobs могут быть звеньями одной цепочки — и продукт нужно проектировать на всю цепочку, а не на одну.

#### Разрыв 2: Нет типизации работ

AJTBD выделяет 7 типов работ, каждый определяет стратегию:

| Тип | Что даёт пайплайну |
|---|---|
| Частотные | Оценка retention и подписочной модели |
| Последовательные | Где именно в цепочке ставить продукт |
| Веральные | Каналы органического роста (критично для solo dev без бюджета) |
| Ориентационные | Как люди узнают о решениях — ключ к дистрибуции |
| Налоговые | Работы из-за поломки решения — источник сильного спроса |
| Обычные | Базовая линия — нет раздражения, сложнее продавать |
| Ложные | Уже фильтруются backward-looking подходом |

Особенно важны **веральные работы** — для solo dev без маркетингового бюджета это единственный путь к масштабу.

#### Разрыв 3: Нет механик создания ценности

Замесин описывает 17+ механик (убить работу, забрать выполнение у человека, уменьшить промежутки, выполнять следующую работу в КПР, выполнять ориентационную работу, и т.д.). Проект обнаруживает работы, но не применяет механики систематически. Гипотеза ценности формулируется «от головы», а не через формализованный инструмент.

#### Разрыв 4: Ориентационные работы и consideration set

Проект фиксирует switching behavior (что менял, почему), но не фиксирует:
- как человек узнал о текущем решении
- что входит в его consideration set
- какие ориентационные работы выполнял перед выбором

Для solo dev без бюджета каналы обнаружения решений — стратегическая информация. Если люди узнают о workaround'ах через Reddit — значит Reddit = канал дистрибуции.

#### Разрыв 5: Критерии ожидаемого результата

Severity и Frequency — характеристики работы. Но разные люди выполняют одну работу с разными критериями (быстро, дёшево, надёжно, статусно, к сроку). Добавление критериев уточнит, для какого подсегмента строить продукт.

#### Разрыв 6: Проектирование предлагаемого графа

Замесин: создатель решения предлагает граф, клиент соглашается или нет. Проект обнаруживает текущий граф клиента, но не проектирует граф, который продукт предлагает. R5 Synthesis мог бы включать: «вот какой граф мы предлагаем и почему он энергоэффективнее текущего».
